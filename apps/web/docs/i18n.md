# 国際化 (i18n) 運用ガイド — swim-hub

Issue #32 Phase 1-A で導入した next-intl 基盤の運用ドキュメント。
将来 Phase 2 以降で 8 言語追加する際の手順をここに集約する。

## 構成サマリー

| 項目 | 値 |
|---|---|
| ライブラリ | `next-intl` v3 |
| URL 戦略 | `localePrefix: 'always'` (`/ja/...`, `/en/...`) |
| ロケール ID | BCP 47 (現状: `ja`, `en`。将来 `zh-CN`, `pt-BR` 等) |
| デフォルトロケール | `ja` |
| 自動ロケール検出 | **無効** (`localeDetection: false` — Googlebot 対策) |
| 翻訳ファイル | `apps/web/messages/{locale}.json` |
| SSOT (Phase 1) | `messages/ja.json` |
| 動的 import | `getRequestConfig` で `import('../messages/${locale}.json')` |
| 型生成 | `apps/web/types/i18n.d.ts` で `Messages` 型を declare |

## ファイル配置

```
apps/web/
├── i18n/
│   ├── routing.ts        # locales / defaultLocale / localePrefix の定義
│   ├── request.ts        # getRequestConfig (Server Components 用)
│   └── navigation.ts     # createNavigation の出力 (Link / useRouter / usePathname / redirect)
├── messages/
│   ├── ja.json           # 日本語 (SSOT)
│   └── en.json           # 英語
├── types/
│   └── i18n.d.ts         # Messages 型 declare (typeof messages.ja.json)
├── middleware.ts         # next-intl middleware → updateSession → CSP の chain
├── app/
│   ├── layout.tsx        # 最小化 (return children のみ)
│   ├── not-found.tsx     # 自前で <html><body> 出力 (root layout に依存できない)
│   └── [locale]/
│       ├── layout.tsx    # <html lang> + Provider 群 + generateMetadata (hreflang)
│       └── ...           # 旧 (authenticated) / (unauthenticated) / (dashboard) を移動
└── components/ui/
    └── LanguageSwitcher.tsx
```

## 翻訳キーを追加する手順

1. **`messages/ja.json` に新しいキーを追加** (SSOT を先に書く)

   ```json
   {
     "common": {
       "appName": "SwimHub",
       "loading": "読み込み中...",
       "error": "エラーが発生しました",
       "save": "保存"          // ← 追加
     }
   }
   ```

2. **`messages/en.json` に同じキー構造で英訳を追加**

   ```json
   {
     "common": {
       "save": "Save"
     }
   }
   ```

3. **TypeScript 型は自動で更新される**

   `types/i18n.d.ts` が `typeof import('../messages/ja.json')` を `Messages` 型として
   declare しているため、`useTranslations('common')` の引数候補や `t('save')` の返り値型は
   自動で更新される。tsc が typo を検出する。

4. **コード側で使用**

   ```tsx
   // Server Component
   import { getTranslations } from 'next-intl/server';
   const t = await getTranslations('common');
   return <button>{t('save')}</button>;

   // Client Component
   'use client';
   import { useTranslations } from 'next-intl';
   const t = useTranslations('common');
   return <button>{t('save')}</button>;
   ```

## ロケールを追加する手順 (Phase 2 以降)

1. **`apps/web/i18n/routing.ts` に追加**

   ```ts
   export const routing = defineRouting({
     locales: ["ja", "en", "de", "zh-CN"],   // 追加
     defaultLocale: "ja",
     localePrefix: "always",
     localeDetection: false,
   });
   ```

2. **`messages/{locale}.json` を作成** (`ja.json` をコピー → 翻訳)

3. **`app/[locale]/layout.tsx` の `generateMetadata` に hreflang を追加**

   ```ts
   alternates: {
     languages: {
       ja: `${SITE_URL}/ja`,
       en: `${SITE_URL}/en`,
       de: `${SITE_URL}/de`,        // 追加
       "zh-CN": `${SITE_URL}/zh-CN`, // 追加
       "x-default": `${SITE_URL}/ja`,
     },
   }
   ```

4. **`components/ui/LanguageSwitcher.tsx` の `LOCALES` 配列に追加**

   ```ts
   const LOCALES = [
     { code: "ja", label: "JA", aria: "日本語" },
     { code: "en", label: "EN", aria: "English" },
     { code: "de", label: "DE", aria: "Deutsch" },     // 追加
     { code: "zh-CN", label: "中", aria: "简体中文" },  // 追加
   ] as const;
   ```

5. **`lib/supabase-auth/middleware.ts` の `SUPPORTED_LOCALES` を更新**

6. **TS 型は自動更新** (新しいロケールの JSON のキー構造が既存と一致する前提)

7. **CI で「全ロケールに同じキーが揃っているか」を検査** (将来追加予定の lint script)

## URL ルーティングの挙動

| アクセス URL | 挙動 |
|---|---|
| `/dashboard` | 308 → `/ja/dashboard` (next-intl middleware) |
| `/ja/dashboard` | そのままレンダリング |
| `/en/dashboard` | そのままレンダリング |
| `/zh/dashboard` | 未定義ロケールのため `notFound()` (next-intl が判定) |
| `/api/*` | middleware の matcher から除外 |
| 静的アセット | matcher から除外 |
| Accept-Language: en | **無視される** (`localeDetection: false`) |

## SEO

- 各ロケール layout で `<link rel="alternate" hreflang>` を出力 (`generateMetadata` の `alternates.languages`)
- `x-default` は `ja` を指す (主市場)
- canonical は **自身のロケール URL** に設定 (`/ja` の canonical は `/ja`、`/en` の canonical は `/en`)
- sitemap.ts は app 直下に維持 (Phase 2 以降で各ロケール URL を `<xhtml:link>` で相互参照)

## middleware chain 順 (重要)

```
1. next-intl middleware (locale 判定 + リダイレクト)
2. updateSession (Supabase 認証ガード)
3. CSP / セキュリティヘッダー付与 (必ず最後)
```

next-intl がリダイレクト (308) を返した場合も、CSP ヘッダーは必ず付与してから return する。

## Cloudflare Workers 制約

- 翻訳 JSON は `getRequestConfig` 内の **動的 import** で chunk 分割
  (`import('../messages/${locale}.json')`)
- ビルド時に各ロケール JSON が独立 chunk として出力される
- 全ロケール JSON を bundle に積まないため、Worker サイズ上限 (有料 10MB) に影響しにくい
- 検証方法: `pnpm build` 後に `.next/` 配下を確認し、`ja.json` / `en.json` が個別 chunk で
  出力されていること

## 既知の制約 (Phase 1-A 時点)

- Phase 1-A は **基盤導入のみ**。既存の日本語ハードコード文字列は翻訳キー化されていない
- `messages/{locale}.json` は最小 3 キー (`common.appName` / `common.loading` / `common.error`) のみ
- 翻訳キー化は **Phase 1-B 以降** で `(unauthenticated)` 配下から段階的に進める
- RTL (アラビア語等) は対象外。新規コードのみ Tailwind の論理プロパティ (`ms-*` / `me-*`) を推奨
